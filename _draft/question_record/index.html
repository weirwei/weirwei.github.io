什么是悲观锁，什么是乐观锁

乐观锁是发生冲突时进行回滚、重试操作，适用于读多写少的场景 悲观锁是发生冲突时阻塞，适用于写多场景

MySQL什么时候进行当前读，什么时候进行快照读

不同的隔离机制有不同的读取方式 RU的时候是当前读，当前读会加锁 RC或RR的时候是快照读， 快照读不会加锁

什么是间隙锁

间隙锁是在执行范围查询的时候对两个索引值之间的空间加锁，可以防止幻读

间隙锁的使用场景

  1. 进行范围查询的时候，如：<、>、between
  2. 有唯一索引插入数据
  • 在RR 场景下，间隙锁会持续整个事务，使用间隙锁可以解决幻读的问题
  • 在RC 场景下，间隙锁持续一条语句

因此，在RC 下，并发性能更好

innodb 执行查询语句的具体步骤：

  1. 查询事务id，获取事务版本号
  2. 通过事务id获取ReadView
  3. 查询存储的数据，将其事务id和ReadView 中的事务版本号进行比较
  4. 不符合ReadView 的可见性规则，则读取undo log 中的历史快照数据
  5. 找到当前事务能够读取的数据返回

ReadView 的内容

  • creator_trx_id: 创建当前ReadView 的事务id
  • m_ids: 当前系统中所有活跃事务（开启但未提交）的id
  • m_low_limit_id: 在生成ReadView 时，当前系统中活跃的读写事务中最小的事务id，即m_ids中最小值
  • m_up_limit_id: 当前系统中事务的id值最大的那个事务id值加1，也就是下一个要生成的事务id

示例:

时间事务A事务B
1开始事务
2开始事务
3第一次查询：select where id = 1
4update where id = 1
5提交事务
6第二次查询：select where id = 1
7提交事务

对比MySQL 加字段和es 加字段

MySQL

  1. MySQL 中的数据页通常以页的形式进行管理和存储。 每个数据页中存储了多行数据，并且这些行数据按照插入的顺序组成一个链表结构。

    当添加字段时，如果要将新的字段插入到已有的数据页中，可能会导致数据页的重新组织和调整，以便将新字段的数据正确地插入到链表中。

  2. 当添加字段时，MySQL 需要确保数据的一致性。

    这意味着在添加字段的过程中，可能需要对表进行锁定，以防止其他并发操作对数据的读写干扰。

    为了避免数据丢失或损坏，MySQL 需要进行数据迁移和重组，以确保添加字段后的数据正确地存储和访问。

Elasticsearch

  1. 采用倒排索引。添加字段就是在倒排索引添加一个新的索引字段，不对原有数据产生影响。
  2. es 的数据以文档形式存储，支持每个文档有不同的字段。新加字段，不改变原有数据，只影响新增的数据。

MySQL 是以数据页的形式进行管理和存储的，那么对于变长 varchar、varbinary、blob、text等字段，它是如何存储的？

https://dev.mysql.com/doc/refman/5.7/en/innodb-row-format.html#innodb-row-format-dynamic

  • 可以在溢出页存储变长字段的数据，通过20字节长度的指针。
  • 大于或等于 768 字节的定长字段，也会被编码成变长字段。
  • 当行数据的长度超过 页大小/2 时，会选择最长的列存储在溢出页。小于等于40字节的 text 和 blob 列会存储在行中。
  • 溢出列会以链表链接的形式存储在多个溢出页上

MySQL有4种行格式：REDUNDANT、COMPACT、DYNAMIC、COMPRESSED

  • Redundant：行格式会把变长字段值的前768字节存在B-Tree的索引记录中，若字段超出长度，则其剩余数据被放在溢出页进行存储。
  • Compact：行格式在溢出页的处理上和redundant行格式基本一致，也是将超过768字节后的变长列放至溢出页上。
  • Dynamic：行格式和Compact行格式有类似的行存储格式（有变长列表等），即会将大变长列完全存储在off-page上，聚集索引上只含有20字节的指针。
  • Compressed：行格式在dynamic行格式基础上增加了压缩特性，对于off-page列的处理模式与Dynamic行格式基本一致。

溢出页的优缺点

优点：

  • 更灵活。使变长数据不受页大小限制。
  • 性能。避免频繁进行数据移动和页分裂。

缺点：

  • 额外I/O。需要读写溢出页的数据。因此，对于可能溢出的列，用不到就不要查了。

什么是页分裂

使用非自增主键的时候，每次插入新的数据，可能会插入到现有数据页中间的某个位置，这样会移动其他数据来满足新数据的插入

执行一条SQL发生了什么

  1. 客户端发送SQL到MySQL服务端。
  2. 【连接器】身份验证：检验客户端权限。
  3. 【解析器】语义分析：在语义分析阶段，检查SQL语句中的表、列是否存在，用户是否有权限。
  4. 【查询优化器】优化SQL：选择最佳执行计划。如索引选择、表连接顺序优化。
  5. 【执行器】执行计划生成：根据查询优化的结果生成执行计划树。指定了SQL的执行顺序和方式。
  6. 【存储引擎】执行SQL：按照执行计划树执行SQL。读取、写入、过滤、排序等。
  7. 【连机器】返回结果：将执行结果返回给客户端。

索引失效的场景

  1. 未遵循前缀匹配原则
  2. 对索引列进行函数操作
  3. 计算导致索引失效，如 where id + 1 = 66
  4. 类型转换，如 where convert(id,char) = '66', where name = 1(隐式转换，name 为字符串类型)
  5. 使用范围查询，如 <
  6. 使用不等于（!= 或 <>）导致索引失效
  7. 使用 OR 操作符，如果其中一个条件无法使用索引，会导致索引失效
  8. 使用模糊查询。使用%进行模糊查询的时候，可能使索引失效。如：%匹配%，%匹配。匹配% 不会失效。
  9. 数据量过小：当表的数据量非常小的时候，会直接全表扫描。
  10. 更新频繁：表的频繁更新会使索引失效。

MySQL自增主键一定是连续的吗

不一定

  1. 自增初始值和自增步长设置不为 1
  2. 唯一键冲突
  3. 事务回滚
  4. 批量插入（如 insert…select 语句）

MySQL的varchar字段为什么常用255

  1. innodb 引擎 在MySQL 5.6 及以前版本中，索引单字段只支持 767 bytes，utf-8 是 3 bytes，所以 255 * 3 = 765 < 767。
  2. utf-8mb4 是 4 bytes。
  3. MySQL5.7 改成 3072 bytes，所以 255 是历史原因。
