记一次上亿级别的数据迁移

前期准备

需要考虑的几个点：

  • 迁移失败。迁移过程中发生报错该怎么处理？
  • 数据正确性。如何验证数据正确性。
  • 业务中断。迁移到一半，因不可抗力因素（服务重启等）中断。
  • 数据源切换。如何切换数据源，使用户无感知。

解决：

  • 迁移失败。数据迁移不要吝啬日志，打印充分的关键日志，方便定位问题。需要准备一个应对失败的方案，如：遇到迁移报错的步骤，是跳过还是中断，后续如何修复这部分错误。
  • 数据正确性。可以通过查询错误日志、统计日志数量、抽样观察等方式。还可以使用双读的方式，对迁移前后的数据进行一致性校验，如果不一致，则报错。
  • 业务中断。保证数据幂等，可重入，使业务重启后可以接着继续任务。
  • 数据源切换。采用双读双写策略。

难点

  1. 分表字段。重点，需要前期梳理好相关的业务，主要从查询、写入方向梳理，判断分表后是否会有无法查询或者无法写入的情况。

笔者在做分表的时候遇到了一个难点场景： 数据表有：用户内容表（用户id、设备id、内容id、内容信息）、用户收藏表（用户id、内容id）、用户生产记录表（用户id、设备id、内容id） 前期迭代的时候，PM要求把用户生产记录绑定设备号，换设备清空，切换账号保留（为什么不记在本地呢，因为PM希望在数据库留一份数据，做数据分析）

流程

  1. 最小分组：1000
  2. 查询文章表（过滤version为-1的数据）
  3. 根据cuid分批
  4. 查询sessionRecord
  5. 根据sessionType 分组（一键成文还是ai写作）
  6. sid 去重，保留最新的
  7. ai写作的文章段落写入 分段写作段落表
  8. 写入新的文章表（如果session大于2048，则写入bigdata）

数据一致性

双读双写要有开关能配置

双读

  • 读数据的时候同时查询两个表的数据
  • 使用diff 工具判断数据是否一致，如果不一致，打ERROR 日志

双写

  • 写数据的时候同时写两个表的数据，使用事务，保证一致性
  • 老表的version字段记-1，迁移数据的时候遇到-1就跳过
  • 双读后需要验证到新数据，并且切换后方便无痕

整体流程

  1. 代码上线
  2. 开启双写
  3. 数据迁移
  4. 开启双读
  5. 持续观察日志
  6. 下线旧表和双读双写逻辑

项目总结

回顾目标和结果

主要目标：

  1. 文章表数据量过多，压力过大，分表缓解数据库查询压力
  2. 分表后确保数据的一致性（双读的diff很重要）
  3. 文章表分表之后，有部分逻辑需要改动（因为分表后的查询依赖分表参数）

次要目标：

  1. 原文章表缺少一些参数，查询链路复杂。需要补充参数
  2. 优化查询方式，将比较大的内容字段拆分，记入段落表，文章表的内容不记全文，该记段落id
  3. 分段成文没有记录段落，需要在迁移时往段落表记一份

关键问题和挑战

挑战

  1. 文章表依赖业务多且复杂
  2. 数据量庞大
  3. 部分逻辑链路长，导致迁移时间会比较长，需要针对迁移性能做优化

设计方案回顾

  1. 没有考虑迁移期间数据更新的情况
  2. 没有明确双读双写的切换逻辑 方案确定了双读双写的方式，但是没有确定好细节，如：
    • 切换双读双写以及从双读双写切回新表的时机
    • 双读双写在哪一层进行（需要考虑修改成本，保证能够做到一键切换）

反思和总结

双读双写review不通过

期望：在双读的时候，diff通过使用新数据，diff不通过使用旧数据

实际：只对新数据进行diff，在diff不通过的时候打印日志。没有在diff通过的时候使用新数据。原因是按照原本diff方案，改动成本过高

反思：

  1. 遇到问题应该及时沟通
  2. 数据的可靠性和一致性的核心目标，不得损害

双读问题记录

  1. all
    • 批量查的排序问题 【使用map】
    • 在迁移的过程中，忽略了sessionRecord不存在的数据 【在文章查不到的时候，返回“文章已下线”的文案】
    • 文章结果接口没传 cuid 【一键成文分享，图片分享失败的情况下，会分享链接】
  2. session 不一致（session不为空）
    • 数字类型，在类型转换的过程中，整型变成了浮点型 【不解决，diff对长度不严格校验】
    • 字段的空值问题 【新表增加 afterFind 逻辑，为空或者null的时候返回 {} 】
  3. articleId 不一致
    • 同一个sid生成篇文章 【批量查的时候不校验 articleId】
  4. 长度不一致
    • 主从同步导致 【不解决】
  5. title 不一致
    • 更新文章结果的时候，会把空标题更新为“无标题” 【更新文章的时候把title也更新】
  6. type 不一致
    • FE sdk bug，导致 sid 重复使用 【不解决】
  7. content 解析失败
    • 算法bug，每个字都会作为一段进行生成，导致 content 长度超过限制 【在content 比较长的时候对content 进行压缩】
  • 在迁移的过程中，忽略了sessionRecord不存在的数据 【在文章查不到的时候，返回“文章已下线”的文案】
  • 文章结果接口没传 cuid 【一键成文分享，图片分享失败的情况下，会分享链接】
