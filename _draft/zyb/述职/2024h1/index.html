目标回顾

O1: 光速写作业务支持

  • kr1: 支持光速写作业务迭代，提高迭代效率，能够快速响应产品提出的需求；
  • kr2: 保证项目的交付质量，因研发自身问题导致的项目延期不超过3%，bug数<总人天数；
  • kr3: 了解直属上游的逻辑，做到方案设计合理无卡点，不做无效的重复工作；
  • kr4: 做好服务总结。对光速整体服务进行一个拓扑梳理，梳理好服务现状，找出服务现有的问题，并寻找优化点及可行优化方案；

自评

  • kr1：支持光速业务和主端语文作文业务的开发和设计，能快速响应需求并设计合适的方案；
  • kr2：代码高质量交付。大多数需求都能做到按时低bug数完成，个别大项目bug数较多，如：议论文讲解。对进度/bug数异常的项目进行总结；
  • kr3：对接上游的时候，会了解教上游服务的逻辑，对于重要的部分会直接申请代码仓库阅读代码。设计方案的后会主动跟响应的对接人提前沟通，保证方案的可行性；
  • kr4：完成对写作助手消息格式的梳理，记录开发过程中发现的问题。但是对光速整体的服务梳理不够完整；

O2：光速写作稳定性建设

  • kr1: 完善监控建设，能够在分钟级别发现并响应线上问题，并且能够在20分钟内解决，不出现故障而感知不到的情况；
  • kr2: 清楚团队的容灾建设（限流和熔断），做到服务触发容灾后，能够快速定位原因，并判断容灾是否有效；
  • kr3: 建设业务监控。根据负责服务的实际业务场景，搭建符合业务需求的场景监控；
  • kr4: 降低服务的错误率，单个服务每天线上的非依赖错误数不高于20数；

O3：个人能力提升

  • kr1: push一个对团队有收益的项目；
  • kr2: 对大项目进行项目总结。主要包括：回顾目标和结果、关键问题和挑战、设计方案回顾、反思和总结等。能够独立搞定有深度的项目；
  • kr3: 积极进行AIGC 实践。探索市面上各类AIGC产品，结合当前我们已有的能力和功能，作出有价值的尝试和探索；

自评

  • kr1：代码lint工具（支持自定义配置，更加符合团队现在） 和 gorm 生成器。但是尚未完成；
  • kr2：对于大项目有文档沉淀，在后续的设计方案和需求开发中能体现出进步，如：议论文讲解暴露了数据库更新字段耗时长，没有提前预估的问题，在新的需求的sql改动中，标注了sql的预计执行时间；
  • kr3：多赛道探索aigc实现，如：https://wegic.ai 、https://lamucal.ai/ 等；

项目记录与总结

文章表分表

背景：文章表数据量过大（2亿+），需要拆表。

目标：

  1. 文章表已cuid维度拆成100个表；
  2. 数据迁移到新表。

项目重点：

  1. 影响面梳理。文章表是光速业务最基础的数据结构，影响的业务面非常广；
  2. 数据迁移方案。需要考虑数据一致性、业务中断、迁移期间的数据变化；
  3. 上线&测试方案（刷数据实际、数据切换方案）；

成果与经验教训：

  1. 成功将文章表按照 cuid 维度拆分成 100 个表，并完成数据迁移；
  2. 明确项目的主要诉求，避免在无关的任务上花费过多精力。这个需求的主要目的是拆表，我却花费尽力去优化表结构，导致项目复杂度上升和延期；
  3. 在新老表切换时，设计好切换的开关。该设计不合理，review的时候发现重构，也是延期的一个因素；

写作助手四期

背景：为了体现差异化，新增「写作导师」功能，使用定制化的 prompt 进行写作指导；

目标：

  1. 基于原本的聊天模式，开辟单独session的聊天会话（智能体）；
  2. 「写作导师」的入口内嵌于主聊天；

项目重点：

  1. 设计主聊天的「写作导师」入口的数据结构。入口有：快捷入口、配置模板、历史消息入口，点击入口需要执行不同的动作，同时需要兼容现有数据结构；
  2. session 切换机制重构。新增「写作导师」后，独占一个 session 的特性与原本主聊天的 session 切换机制有些许冲突，需要局部重构；

成果与经验教训：

  1. 成功在原本的聊天模式中开辟了单独 session 的聊天会话，并将「写作导师」的入口内嵌于主聊天；
  2. 重构了 session 切换机制，解决了新增「写作导师」后与原本主聊天的 session 切换机制的冲突；

作文大赛 背景：为了提升主端语文作文模块影响力，开办作文大赛活动。 目标：

  1. 用户可自主投稿参与活动；
  2. 活动有排行榜，不同排行榜排名的计算方式不同；
  3. 支持搜索；
  4. 支持作品投票；
  5. 支持给用户push活动信息；

项目重点：

  1. 数据结构设计。全新的流程，需要存储活动数据，关联主端作文数据；
  2. 多端交互流程。依赖主端投稿、作文等功能，保证数据可达；
  3. 排行榜设计。排行榜依赖投票、浏览量、投稿时间等因素；

成果与经验教训：

  1. 成功实现用户自主投稿、排行榜展示、搜索、投票和推送活动信息等功能；
  2. 数据设计合理，能够有效存储活动数据，并兼容后续可能举行的作文大赛活动；
  3. 排行榜设计灵活，可以通过后台宏观调控；
  4. 项目中实现了很多缓存、排行榜相关的逻辑，使我对响应的业务场景和缓存的使用更得心应手；
  5. 大型项目完成后要多注意线上监控和日志，特别是在业务高峰期的时候，可能会遇到一些高峰期才会暴露的问题。

数据库拆集群 背景：writing 集群磁盘压力过大，需要紧急处理磁盘容量 目标：

  1. 临时删除一部分数据；
  2. 申请新的集群，把数据迁移过去，并且不影响线上功能；

项目重点：

  1. 分析迁移数据。哪些数据适合分集群访问，该如何访问；
  2. 新增数据源开关控制。使用redis记录可切换的表下标；
  3. 数据同步方案。通过dts同步数据；
  4. 检查数据迁移情况。数据库监控&查询数据；
  5. 删除旧表，下线开关代码；

成果与经验教训：

  1. 成功缓解了MySQL集群的磁盘压力，确保系统的稳定性；
  2. 在项目生产过程中，要密切关注业务的监控；

议论文讲解 背景：AI写作扩展议论文场景，适配高中学段。 目标：

  1. 实现议论文流程：审题、立意、框架、素材、成文；
  2. 接入llm；
  3. 实现拍照议论文；

项目重点：

  1. 接入llm。在光速侧调用大模型，自定义解析函数；
  2. 议论文流程数据基于现流程的写作数据结构进行存储；
  3. 流程模块化设计，各个流程结构解耦，互相独立；
  4. 监控建设。对于llm，自定义业务监控看板；

成果与经验教训：

  1. 成功实现了议论文的流程，包括审题、立意、框架、素材和成文；
  2. 顺利接入了 LLM，并在光速侧调用大模型，自定义了解析函数，请求稳定，且数据解析准确；
  3. 建立了针对 LLM 的自定义业务监控看板；
  4. LLM 输出结果具有不可确定性，需要建立一系列长期维护的单测case用例；
  5. 方案设计的时候就需要考虑上线各个环节的耗时。上线时没有预判好数据库执行时间，为了不影响上线，使用临时方案解决；

成果与亮点：

  1. 能够独立承担一些较大的项目，并关注项目全流程；
  2. 能够提出符合当前业务架构的设计方案，并实施；
  3. 积累了丰富的经验教训，如明确项目诉求、关注项目核心诉求、注意线上监控、完善设计方案等。

问题与不足：

  1. 在项目中存在一些延期的情况。需要进一步优化项目管理和时间安排，在方案设计的时候，使用伪代码的方式合理评估项目复杂度；
  2. 在方案设计中有一些过度设计和疏漏的情况，需要时刻回顾项目的主要目标，并结合排期在方案设计阶段，和产品做好需求的确认和方案的研讨；
  3. 技术调研的深入有待提高。工作的空隙会关注当前市场上aigc的产品，有时会有一些想法，但是缺少深入的实践。

个人成长：

  1. 在大需求的设计和开发上的得到了较充分的实践；
  2. 在方案的设计上，能够平衡时间和内容，做到兼顾重点，并在合理的时间内完成；
  3. 性格上更加稳重，对于突发的问题和事件，能够从容应对。
